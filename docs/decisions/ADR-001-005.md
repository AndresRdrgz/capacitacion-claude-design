> Formato: ADR (Architecture Decision Record). Una decisión por archivo. Inmutable una vez cerrado.

## ADR-001: No crear plataforma nueva, extender iAcademy

**Fecha**: 2026-06-30
**Status**: Aceptada

### Contexto

El usuario pidió construir una "plataforma de aprendizaje online sobre Claude" partiendo de un deck estático de 4 slides. La intuición inicial era construir desde cero con Next.js + Supabase + Clerk + Stripe.

Al investigar, descubrimos que iAcademy ya tiene una plataforma e-learning completa: Django 5.1 + DRF + Next.js 16 + Vimeo Pro + JWT + Postgres, con cursos, lecciones, quizzes, gamification, comentarios, certificados, y 11 admin ViewSets para gestionar todo.

### Decisión

El MVP **no es una plataforma nueva**. Es una extensión de la plataforma iAcademy existente, en el mismo monorepo.

### Consecuencias

**Positivas**:
- Reutilización: auth, progreso, gamification, comentarios, certificados
- Continuidad operativa: mismo deploy, mismo monitoring, mismo equipo
- Costo: ~$20/mes (Vimeo) vs $150+/mes de infra nueva
- Velocidad: 4 sprints en vez de 12+

**Negativas**:
- Restricciones del schema legacy (e.g. `OneToOneField` en `categoria` que hay que migrar)
- Deuda técnica heredada (e.g. el auto-sync de grupos es frágil)
- Limitaciones de UX: hay que encajar en el patrón visual de iAcademy

**Neutras**:
- La marca "iAcademy" se mantiene (decisión consciente de Andrés)

### Alternativas consideradas

- **Plataforma nueva (Next + Supabase + Clerk + Stripe + Bunny.net)**: rechazo, es 4x más trabajo, 3x más costo, y duplica funcionalidad
- **Sub-app dentro de iAcademy pero con stack nuevo (Next + Vercel separado)**: rechazo, genera deuda de integración

## ADR-002: Sin pagos en el MVP

**Fecha**: 2026-06-30
**Status**: Aceptada

### Contexto

Andrés confirmó que el acceso al curso se controla mediante el sistema de matrículas existente en iAcademy (legacy `Inscripcion`). No hay intención de agregar pasarela de pagos en el MVP.

### Decisión

El MVP **no implementa Stripe, Yappy, ACH ni ninguna pasarela de pago**. El acceso es manual: cuando un estudiante paga en iAcademy, se crea una `Inscripcion` legacy → signal crea `GrupoMember` → signal crea `CourseEnrollment` → el estudiante ve el curso en su dashboard.

### Consecuencias

**Positivas**:
- MVP más simple y rápido
- Sin costos de pasarela
- Sin compliance PCI

**Negativas**:
- El "MVP" no es realmente un producto self-serve — requiere intervención manual
- No se puede medir la conversión a pagos

**Mitigación**:
- El schema ya contempla `access_tier` para que agregar Stripe en el futuro sea un cambio de sprint 2-3, no de mes 6
- Las decisiones de pricing están documentadas en `.plan/research/mercado-plataformas-edtech-panama.md` para cuando se agregue

### Alternativas consideradas

- **Stripe + Yappy desde el día 1**: rechazo, agrega 2-3 sprints y dependencias externas
- **Stripe solo (sin Yappy)**: rechazo parcial, Yappy es el método dominante en Panamá, excluirlo es perder mercado

## ADR-003: Contenido creado vía admin, no hardcoded

**Fecha**: 2026-06-30
**Status**: Aceptada

### Contexto

iAcademy ya tiene `CourseAdminViewSet`, `SectionAdminViewSet`, `LessonAdminViewSet`, `QuizAdminViewSet`, etc. El deck original son 4 slides hardcoded en código.

### Decisión

El contenido del curso Claude se crea **como data en la DB** vía:
- Django admin para gestión manual
- Data migration (0025) para el seed inicial reproducible

**Nunca** se hardcodea contenido de curso en el código de la app.

### Consecuencias

**Positivas**:
- El equipo de producción de cursos (no-devs) puede crear/editar cursos sin tocar código
- El seed es reproducible (idempotente) y testeable
- Los cursos son data, no código → faciles de versionar, hacer backup, exportar

**Negativas**:
- Hay que escribir el admin UI bien (sprint 1)
- Cambiar el schema requiere migration (más fricción que cambiar un JSON)

### Alternativas consideradas

- **Markdown files en `/content/`**: rechazo, no permite UI rica ni quizzes
- **Headless CMS (Contentful, Sanity)**: rechazo, agrega dependencia externa y costo
- **JSON files + custom loader**: rechazo, anti-patrón Django

## ADR-004: Repositorio de planning separado del código

**Fecha**: 2026-06-30
**Status**: Aceptada

### Contexto

El deck original vive en `capacitacion-claude-design/` (Next.js estático). El código del MVP va a `iAcademy/` (monorepo Django + Next.js). El usuario pidió crear un proyecto de GitHub en el repo del deck.

### Decisión

El repo `capacitacion-claude-design/` se queda como **repositorio de planning/docs del MVP**:

```
capacitacion-claude-design/
├── .plan/                 # ← ESTO es lo nuevo
│   ├── VISION.md
│   ├── ARCHITECTURE.md
│   ├── SPRINT-BACKLOG.md
│   ├── sprints/           # detalle de cada sprint
│   ├── handoffs/          # prompts listos para Claude Code
│   └── research/          # estudios
├── docs/                  # (opcional) para materiales
├── app/                   # el deck original
├── components/            # el deck original
├── public/                # el deck original
└── README.md              # actualizado
```

**El código de la plataforma NO vive aquí.** Vive en `iAcademy/`.

### Consecuencias

**Positivas**:
- Separación clara: planning vs código
- El deck original puede quedar accesible (URL estable)
- Los handoffs para Claude Code están versionados y son revisables

**Negativas**:
- Dos repositorios donde antes había uno
- Hay que comunicar bien la separación al equipo

### Alternativas consideradas

- **Todo en `iAcademy/`**: rechazo, `iAcademy/` es código de producción, no debe tener planning docs
- **Wiki / Notion**: rechazo, queremos docs versionados con git
- **Eliminar `capacitacion-claude-design/`**: rechazo, hay URL desplegada apuntando ahí

## ADR-005: Marca "iAcademy" para el curso Claude

**Fecha**: 2026-06-30
**Status**: Aceptada (pendiente confirmación final de Andrés)

### Contexto

El usuario dijo "el nombre seguirá siendo iAcademy". La marca iAcademy tiene tracción en Panamá como academia presencial.

### Decisión

El curso Claude se lanza bajo la marca **iAcademy**, sin sub-marca separada.

### Consecuencias

**Positivas**:
- Reutilización de marca existente (reconocimiento, confianza)
- No se fragmenta la oferta

**Negativas**:
- El público meta (developers aprendiendo Claude) no es exactamente el público de iAcademy (profesionales tomando cursos presenciales)
- Riesgo de "dilución de marca" si el curso no se ve serio

**Mitigación**:
- Diseñar la landing del curso Claude con look más "tech" (no "academia tradicional")
- Si en 6 meses hay tracción, evaluar sub-marca "iAcademy Tech" o "iAcademy Lab"

### Pendiente

- Confirmar con Andrés antes del sprint 3 (frontend) si la marca funciona así o quiere ajustes
